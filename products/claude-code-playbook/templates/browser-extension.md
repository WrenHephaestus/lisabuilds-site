## Conventions

### Extension Architecture
- **Background (service worker):** Stateless event handler. No long-running processes — Chrome *will* kill it, usually at the worst possible time. Use `chrome.alarms` for periodic work, `chrome.storage` for state.
- **Content scripts:** Run in page context. Minimal footprint. Communicate with background via `chrome.runtime.sendMessage`.
- **Popup/Options/Sidepanel:** Standard UI — React components with chrome.storage for persistence.

### Message Passing Pattern
```typescript
// Define all message types in types.ts
type Message =
  | { type: "GET_DATA"; payload: { url: string } }
  | { type: "SAVE_SETTING"; payload: { key: string; value: any } }

// Background listens and routes
// Content/popup sends via typed helper in messaging.ts
```

### Storage
- Use `chrome.storage.local` for large/frequent data
- Use `chrome.storage.sync` for user settings (syncs across devices, 100KB limit)
- Always wrap in typed helper functions — never call `chrome.storage` directly from UI
- Handle the case where storage is empty (first install)

- Request minimum permissions in manifest
- Use `optional_permissions` for features that don't apply to all users

## Do NOT

- Don't use `chrome.tabs.executeScript` when a declared content script works — it requires broader permissions
- Don't store sensitive data (API keys, tokens) in `chrome.storage` without encryption
- Don't use `alert()`, `confirm()`, or `prompt()` in content scripts — they show as coming from the page
- Don't assume the service worker is always running — it restarts frequently in MV3
- Don't request `<all_urls>` permission unless absolutely necessary — it triggers Chrome Web Store review
